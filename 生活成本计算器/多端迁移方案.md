# 多端迁移方案：Taro/uni-app + JS 核心

> 决策（2026-06-21）：放弃 Flet（编译不出微信小程序）、放弃云函数跑计算（违背离线）、
> 放弃 FastAPI+React 网页版（要服务器、非小程序）。确定走 **跨端框架（Taro/uni-app）+
> 一份 TS 核心 + 云端 JSON 数据**：计算在端侧（离线秒算），数据上云（改一次全端生效）。
> 本文档是执行蓝图，下次开工照此推进。

## 一、目标与原则

- **端优先级**：微信小程序 > Web(H5) > 桌面(Electron 套 Web，替代现 Python exe)。
- **离线优先**：计算全部在端侧跑（TS），断网也能算。这是面向"网络不稳的劳动者"的硬约束。
- **一份核心、多端共用**：计算逻辑和静态数据只维护一份（TS + JSON），UI 各端做（小程序 UI 范式和桌面不同，不可复用）。
- **数据热更新**：各地政策数据（最低工资/失业金/医保报销率/低保…）放云端 JSON，改一次全端生效，免发版。
- **诚实边界**：现有 Python 桌面版（tkinter + calc_engine）**过渡期保留**，最终由 Web/Electron 替代后再退役。

## 二、技术选型

### 跨端框架：Taro vs uni-app（二选一，执行时按团队熟悉度定）

| 维度 | **Taro 4（推荐）** | uni-app |
|---|---|---|
| 语法 | React + TypeScript | Vue 3 + TS |
| 编译目标 | 微信/支付宝/抖音小程序、H5、React Native | 几乎全部小程序、H5、App |
| 与 TS 核心契合 | ✅ React+TS，和核心类型系统统一 | ✅（Vue+TS 也行） |
| 生态/资料 | 京东出品，偏现代，社区活跃 | DCloud 出品，国内案例/文档更多，HBuilderX 工具链集成 |
| 上手 | 熟 React 更顺 | 熟 Vue / 纯新手更顺 |

**倾向 Taro 4 + React + TS**：核心用 TS 写，UI 也用 React+TS，类型系统贯通，翻译 `calc_engine` 时能借类型对齐公式、交叉验证。若更熟 Vue 或想要更多国内抄作业案例，uni-app 同样可行，方案其余部分不变。

### 其余选型

- **核心语言**：TypeScript（类型安全，强制对齐 Python 公式签名）。
- **数据格式**：JSON（从 `cost_data`/`rights_data`/`medical_data`/`relief_data` 导出）。
- **数据托管**：腾讯云 COS（与微信生态近）或七牛，配一个 `version.json` 做版本比对。
- **小程序本地存储**：`wx.setStorageSync`（同步、简单），替代现在的 `profile.json`/`tracking/*.json`。
- **UI 组件库**：Taro 用 NutUI-React-Taro；uni-app 用 uView。
- **状态管理**：轻量即可（Zustand / Pinia / Context，按框架选）。

## 三、架构分层

```
core/                      ← 一份 TS 核心（翻译自 Python，纯逻辑无 UI）
  types.ts                 ← 类型定义（Profile / Result / Metrics…）
  calc/                    ← 纯计算函数（对应 calc_engine）
    situation.ts           ← compute_current_situation / family / risk（含 overrides）
    loan.ts                ← 真实年化 IRR / 还款对比 / 可承受负债 / 雪球雪崩 / 螺旋 / 债务健康
    housing.ts             ← compare_buy_rent / housing_fund / rate_stress / compare_cities
    rights.ts              ← overtime / min_wage / unemployment / subsidy / injury / tax / relief
    medical.ts             ← estimate_inpatient
    milestones.ts          ← 人生三座山
    prompts.ts             ← build_*_prompt（提示词生成，纯函数，含 _profile_brief）
  data/                    ← JSON 数据 + 加载器
    cost.json rights.json medical.json relief.json
    loader.ts              ← 云端拉取 + 本地缓存 + 版本比对
miniprogram/               ← Taro 小程序 UI（页面/组件，调 core）
web/                       ← Web/H5 UI（可选，复用 core）
```

## 四、迁移路径（分阶段，每阶段独立可交付）

### 阶段 0：核心提取与翻译清单（最小风险，对 Python 版也有益）
- 扫描 `calc_engine.py`（~1900 行）：列出所有纯函数、依赖的 `_data` 字段、调用关系，产出 **翻译清单**（函数名/签名/依赖/建议顺序）。
- 把 `cost_data`/`rights_data`/`medical_data`/`relief_data` 的数据结构**导出为 JSON**（写个一次性 Python 脚本 `export_data.py`）。
- 产出物：`翻译清单.md` + `data/*.json`。**这步即使最终不做小程序，也让 Python 版数据源统一**（可改 Python 读 JSON），并给公式提供第二份实现做交叉验证。

### 阶段 1：TS 核心实现 + 交叉验证
- 按翻译清单用 TS 重写 `core/calc/*`，逐模块推进（建议顺序：situation → loan → housing → rights → medical → milestones → prompts）。
- **交叉验证**：复用 Python 版的测试用例（同一组输入，Python 与 TS 输出比对），逐函数对齐。IRR/等额本息/失控检测这些数值敏感的函数重点验。
- `data/loader.ts`：本地内置 JSON（随包）+ 启动拉云端 version + 按需更新 + 本地缓存兜底。

### 阶段 2：数据云端化
- JSON 上传对象存储，发 `version.json`（含各数据文件版本号 + 时间戳）。
- 加载策略：启动读本地缓存版本 → 拉云端 version → 有更新则下载新 JSON → 写本地缓存；离线/失败则用本地缓存（旧版可用，不阻断）。

### 阶段 3：小程序 UI（Taro）
- Taro 脚手架 + TS + NutUI。
- 页面（对齐现有导航）：① 档案 ② 处境 ③ 城市与住房 ④ 三座山 ⑤ 借贷真相 ⑥ 劳动权益 ⑦ 求助与反诈 ⑧ 医保就医 ⑨ 长期跟踪 ⑩ 关于。
- 档案/跟踪用 `wx.setStorageSync` 存（对应 `profile.json`/`tracking/*.json`）。
- 富文本结果：小程序用 `<rich-text>` 或自绘分段着色（对应 Python 的 RichNote 智能着色逻辑）。
- 「按实际改」「问 AI 提示词」「关闭存跟踪快照」等交互按现有逻辑平移。

### 阶段 4：Web/桌面（可选，收尾）
- Taro H5 出网页版，或另起 React 网页。
- 桌面用 Electron 套这个 Web，替代 Python exe，淘汰 Python 桌面版。

## 五、关键决策点（执行前需敲定）

1. **Taro vs uni-app**：见选型表，倾向 Taro+React+TS。
2. **数据托管方**：腾讯云 COS（推荐，微信生态）vs 七牛 vs GitHub Pages（免费但慢）。
3. **Python 桌面版退役时机**：建议 Web/Electron 版功能对齐后退役，过渡期并存。
4. **小程序包大小**：主包 2M 限制，数据 JSON 大（医保/低保各省数据）→ 走分包或纯云端拉取（首启联网，之后缓存）。

## 六、风险与工作量大头

- **`calc_engine` 翻译**：~1900 行纯函数，结构清晰可分模块，配合交叉验证可控。**这是核心工作量但不是最大风险**。
- **`_data` 导出 JSON**：机械活，脚本化（`export_data.py` 遍历模块导出）。
- **小程序 UI 全部重写**：tkinter UI 完全不能复用，10 个页面的交互/布局重画——**这是最大工作量**。可分页推进，先做处境/借贷/权益等高频页。
- **数值一致性**：IRR 二分法、等额本息、个税累进的浮点与边界，务必逐用例对齐 Python 版。
- **小程序限制**：包大小、`rich-text` 能力有限、无文件系统（用 storage 替代）。

## 七、明确不做（已论证排除）

- ❌ **Flet**：底层 Flutter，微信小程序运行时不兼容，编译不出小程序。
- ❌ **云函数跑计算（豆包方案）**：① 微信云开发云函数只收 Node.js，不收 Python（"Python 云函数"不存在，要用得走云托管容器或腾讯云 SCF，更重）；② 把本该本地的计算搬上云，换来永久联网依赖、延迟、冷启动、成本、隐私风险，与"离线优先"冲突。
- ❌ **FastAPI+React+shadcn 网页版**：要服务器、非小程序、与离线诉求不符。
- ✅ **数据上云**保留（云端 JSON 拉取，解决"数据一改全端生效"），但**计算留在端侧**。
